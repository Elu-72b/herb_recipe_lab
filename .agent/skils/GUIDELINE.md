# Herb Recipe Lab 開発規約

## 1. ワークフロー

- 1 Issue = 1 Branch = 1 PR。
- 常に issue.md の順序に従い、現在のタスクに集中する。

## 2. Git規約

- ブランチ: feature/#[issue番号]-[内容]
- コミット: [prefix]: [内容] (feat, fix, docs, style, refactor, test)

## 3. 実装のこだわり

- 単位管理: recipe_herbs の unit を使用。
- 2段階保存: STEP 1（配合）→ STEP 2（飲用ログ）のフローを厳守。
- UI: Stimulusによる動的制御、Turbo Streamによる非同期辞典呼び出し。

## 4. AIへの禁止事項

- 独断でのテーブル削除（マイグレーションは必ず提案後に実行）。
- issue.md にない機能の勝手な追加。
- 同じファイルを何度も読み直す「ループ思考」。

## 5.　処理実行時間の制限

### 実行時間を計測してログに記録

time gemini-cli "Your Prompt" >> gemini_speed.log

### ⏳ 処理時間（5分ルール）に基づく実行戦略

あなたはリクエストを受理した際、回答の生成や処理に**5分（300秒）以上**かかると予測される場合、以下のプロトコルに従って処理を分岐・提案してください。

#### 1. 処理時間の自己見積もり

タスク実行前に、以下の要素から予測時間を算出してください。

- 処理対象のコード行数（1,000行を超えるリファクタリング等）
- ステップ数の多さ（10ステップ以上の複雑な手順）
- 外部APIやCLI経由の重い計算処理

#### 2. 分岐プロトコル

**【A】予測5分未満**：即時実行

- 通常通り処理を実行し、完結した回答を提供してください。

**【B】予測5分以上**：段階的実行（Incremental Strategy）
直接的な回答を中断し、以下の「分割提案」を行ってください。

1. **タスクの分割**: 処理を2〜3つのフェーズに分割することを提案する。
2. **中間アウトプット**: 最初のフェーズの結果のみをまず出力し、ユーザーの確認を待つ。
3. **モデルの変更提案**: 速度重視の「Gemini 1.5 Flash」への切り替えを検討させる。

#### 3. CLI/プログラム側での制限（指示）

ユーザーに対し、以下の実装を推奨または自ら適用してください。

- **タイムアウト設定**:
  クライアント側の設定で `timeout: 300` を明示的にセットするよう促す。
- **ストリーミング利用**:
  レスポンスが長くなる場合は `stream: true` を有効にし、ユーザーが途中で出力を確認できるようにする。

#### 4. 中断判断の基準

以下の状況が発生した場合は、速やかに処理を中断し、代替案を提示してください。

- ループ処理が10回を超えても収束しない場合。
- 1つのコードブロック生成に3分以上を要している場合。
