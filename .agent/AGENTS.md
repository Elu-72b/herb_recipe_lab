# AGENTS.md

このファイルのスコープは `.agent/` 配下です。ここにある設計メモ、Issue 管理、AI 向けガイドを編集するときは、この指示に従ってください。

注記:
- Codex の仕様上、`AGENTS.md` は置かれたディレクトリ配下にだけ適用されます。
- このファイルはリポジトリ全体には効きません。アプリ本体にも同じ方針を適用したい場合は、将来的にリポジトリ直下へ `AGENTS.md` を追加してください。

## 目的

- `.agent/` を「開発判断の一次情報置き場」として保つ。
- 実装前後で参照すべき文書を明確にする。
- AI が Issue の範囲を越えて機能を広げないようにする。

## 参照優先順

`.agent/` 配下を編集するときは、次の順で読みます。

1. `.agent/issue.md`
2. `.agent/skils/GUIDELINE.md`
3. `.agent/skils/GEMINI.md`
4. `.agent/er_diagram.md`
5. `README.md`

判断が衝突した場合:
- ユーザーの最新指示を最優先にする。
- 次に `issue.md` を優先する。
- 実装方針は `GUIDELINE.md` と `GEMINI.md` を参照して整合を取る。

## このリポジトリの前提

- 現在のアプリは初期段階の Rails 7.2 / PostgreSQL 構成。
- `compose.yml` に `web` と `db` の定義がある。
- `Gemfile` は最小構成で、Rails 本体の導入直後に近い状態。
- `.agent/` には MVP Issue、ER 図、AI 補助ガイドが既に存在する。

## `.agent/` 配下で期待する文書

- `issue.md`: 現在の開発順序、未完了タスク、完了条件の管理。
- `er_diagram.md`: データ構造の草案。テーブル追加や関連変更時に更新する。
- `skils/`: AI への作業方針やロール定義。大きな運用変更があるときだけ更新する。
- 新規文書を作る場合も、「何の意思決定を支える文書か」を冒頭に一文で書く。

## 編集ルール

- 既存の Issue 順序を勝手に入れ替えない。必要なら理由を本文に残す。
- まだ確定していない仕様は、断定口調ではなく「案」「仮」「要確認」と明記する。
- 実装済みと未実装を混在させる場合は、チェックボックスや見出しで状態を分離する。
- ER 図と Issue の内容がずれたら、両方を同じ変更で更新する。
- `.agent/skils/` の内容は、実装規約の変更を伴う場合だけ編集する。
- Issue にない機能追加を提案する場合は、まず `issue.md` に候補として追記してから実装側へ進む。

## 実装時の行動指針

- 1 Issue ごとに作業の焦点を絞る。
- Rails では Controller を薄く保ち、複雑な業務ロジックは Model / Service に逃がす前提で設計する。
- MVP では「2段階保存」「unit 管理」「Stimulus/Turbo を使った体験」を壊さない。
- 破壊的な DB 変更や Issue にないスキーマ変更は、文書に根拠を書かずに進めない。

## 更新時のチェック

`.agent/` 配下だけを編集した場合:
- 必須のプログラム実行チェックはありません。
- ただし Markdown の整合性、見出し構成、参照ファイル名は目視で確認してください。

アプリ本体の仕様に影響する更新を `.agent/` に反映した場合:
- 関連する `README.md` や実装ファイルとの差分を確認し、矛盾を残さない。
- 実装側に未着手の内容は「予定」または「未実装」と明記する。

## AGENTS.md のベストプラクティス反映メモ

この叩き台は、OpenAI の Codex 公式案内にある次の方針を踏まえています。

- AGENTS.md には、コードベースの案内方法、標準的な作業手順、テストコマンドを書く。
- スコープは配置ディレクトリ配下に限定される。
- チェックコマンドを書くなら、作業後に実行可能である必要がある。
- Codex は、環境設定・信頼できるテスト・明確な文書があるほど安定して働く。
